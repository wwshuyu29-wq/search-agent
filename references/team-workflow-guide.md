# Search Agent 团队使用说明

这份文档给第一次使用 `search-agent` 的同事看。它不要求你会写代码，只解释三件事：

1. 你应该怎么向 Codex 提需求。
2. Codex 会按什么逻辑推进调研。
3. 哪些节点必须由你确认，AI 不能替你做决定。

`search-agent` 的目标不是“帮你随便搜一堆资料”，而是把一个调研题目变成一条可检查的证据链：先确认需求，再分工检索，再质检来源，再结构化分析，最后生成带引用、可复盘、少 AI 味的报告。

---

## 1. 先理解这条工作流

你可以把它想成一个调研项目组：

- 你是需求方和最终审核人。
- Codex 是项目经理，负责拆任务、派子 Agent、汇总结果。
- 子 Agent 分别负责意图理解、搜索计划、官方来源、媒体来源、RSS 新闻、B站/社媒、金融数据、营销情报、来源质检、结构化分析、引用审计、报告写作和去 AI 味。

最重要的规则是：**没有经过你确认，Codex 不会从 Brief/审核卡片进入正式检索；没有通过来源质检和引用审计，报告不能直接写结论。**

整体流程：

```text
你提出调研题目
    ↓
[T1] 需求理解 & 审核卡片确认
    ↓ 你确认
[T2] 搜索计划 & 多源检索
    ↓ 有冲突时你确认口径
[T3] Source Pack 整理 & 来源质检
    ↓
[T4] 结构化分析 & 专家判断
    ↓
[T5] 报告大纲 / 报告形态确认
    ↓ 可选确认
[T6] 报告成稿
    ↓
[T7] AI 审核 & 引用审计
    ↓ 你提出终审意见
[T8] 修订 & 差异审核
    ↓
[T9] 终稿确认 & 存档
```

> 和普通写作工作流不同，Search Agent 的核心不是“写得像文章”，而是“每个判断都能追到来源，每个来源都知道置信度，每个结论都服务于业务问题”。

---

## 2. 你应该怎么提需求

最简单的写法：

```text
用 search-agent 帮我调研【对象/主题】，
面向【目标读者】，
重点回答【业务问题】，
时间范围是【最近多久/哪个季度/哪一年】，
先输出审核卡片，等我确认后再继续。
```

示例：

```text
用 search-agent 帮我调研高德地图最近三个月上了什么新功能，
面向百度地图市场组，
重点回答这些动作对我们有什么市场启示，
先输出审核卡片，等我确认后再继续。
```

如果你暂时不知道怎么写，也可以只说：

```text
用 search-agent 帮我想一下百度地图市场组最近应该关注哪些竞品动态，
先帮我整理调研方向，不要直接开始搜索。
```

这时 Codex 会先做意图识别和方向澄清，而不是马上联网搜索。

## 3. T1 — 需求理解 & 审核卡片确认

### 你来做

告诉 Codex 你要调研什么。最好包含四个信息：

| 信息 | 含义 | 示例 |
|---|---|---|
| 调研主题 | 要研究谁或什么事 | 高德地图近期新功能 |
| 目标读者 | 报告给谁看 | 百度地图市场组 |
| 输出类型 | 你要报告、简报、方案还是事实速查 | 竞品动态简报 + 市场响应建议 |
| 调性/用途 | 是内部研判、对外发布、领导汇报还是方案讨论 | 内部决策，结论直接一点 |

如果你没有给全，Codex 应该追问，不应提前进入搜索。

### Codex 来做

Intent Router Agent 会先做 LLM 语义理解，判断：

- 研究对象是什么。
- 你真正要支持的业务决策是什么。
- 报告给谁看。
- 时间范围是什么。
- 需要哪些证据。
- 是否需要营销、金融、行业、竞品等专家 skill。

然后 Codex 会输出审核卡片。审核卡片至少要包含：

- 调研主题。
- 调研目的。
- 建议框架或框架组合。
- 拆解维度。
- 中文/英文搜索关键词。
- 信息源范围。
- 拟调用的子 Agent 和专家 skill。
- 仍不确定的问题。

### 会调用哪些 Skill / 工具

| Skill / 工具 | 什么时候用 | 有什么用 |
|---|---|---|
| `marketing-ideas` | 你说“想方向”“不知道从哪看” | 帮你发散可研究角度 |
| `marketing-plan` | 你明确要营销方案/GTM/传播计划 | 帮框定营销计划骨架 |
| `startup-analysis` | 创业公司、VC、融资、加入创业公司类问题 | 帮判断创业/投资视角 |
| `yfinance-data` / `funda-data` | 单一金融数字或严肃金融分析 | 快速拿结构化金融数据 |
| `intent_classifier.py` | LLM 判断之后 | 作为规则校验和 CLI 兜底 |
| `framework_combinator.py` | 确认核心问题后 | 组合最小可用框架，避免堆框架 |

### 产出 Artifact

- `IntentBrief`：Codex 对需求的结构化理解。
- `AuditCard`：给你确认的审核卡片。

### 进入下一步条件

你明确回复：

```text
确认
```

或提出修改：

```text
框架改成 3C + JTBD，时间范围改成最近半年，UGC 只看 B站和小红书。
```

**闸门：你不确认，Codex 不进入正式检索。**

---

## 4. T2 — 搜索计划 & 多源检索

### 你来做

通常不用操作。只有在以下情况需要你介入：

- 你临时补充内部资料。
- 你要求跳过某类来源。
- Codex 发现关键来源缺失，需要你确认是否继续。

### Codex 来做

Search Planner Agent 会把确认后的审核卡片拆成具体检索任务：

- 每个框架维度要搜什么。
- 哪些任务查官方来源。
- 哪些任务查媒体。
- 哪些任务查 RSS/新闻。
- 哪些任务查 B站/社媒。
- 哪些任务需要金融数据。
- 哪些任务需要营销情报 skill。

然后 Source Hunter 类子 Agent 并行工作。

### 子 Agent 分工

| 子 Agent | 负责什么 | 不能做什么 |
|---|---|---|
| Official Source Hunter | 官网、公告、IR、监管披露、应用商店更新 | 不能把媒体报道当官方事实 |
| Media Source Hunter | 中文媒体、英文深度文章、行业分析 | 不能让媒体观点覆盖官方事实 |
| RSS/News Hunter | 快讯、RSS、近期新闻信号 | 不能把 `relevance_score` 当真伪分数 |
| UGC/Social Hunter | B站、微博、小红书、知乎、Reddit 等公开讨论 | 不能把单条评论概括成市场事实 |
| Finance Data Hunter | 财报、股价、市值、估值、结构化金融数据 | 不能给投资建议 |
| Marketing Intelligence Hunter | 竞品定位、用户、渠道、营销方法论 | 不能把通用营销建议当事实来源 |

### 会调用哪些 Skill / 工具

| Skill / 工具 | 什么时候用 | 有什么用 |
|---|---|---|
| Firecrawl | 官方页、深度媒体、JS 渲染网页 | 搜索和抓正文 |
| realtime-search / 百度搜索 | 中英文通用搜索 | 找中文媒体、官网、公开网页 |
| finance-rss-reader | 近期财经/产业/RSS 新闻 | 快速扫新闻信号 |
| B站 CLI / agent-reach / opencli | UGC 和社交平台 | 抓公开讨论和视频信息 |
| yfinance / Funda 等 | 金融数据任务 | 拿行情、财报、估值等结构化数据 |
| marketing skill catalog | 营销/竞品/用户任务 | 选择合适的营销分析方法 |

### 产出 Artifact

- `SearchPlan`：检索任务清单。
- `SourceListFragment`：每个 Source Hunter 找到的来源片段。

每条来源要有：

- `source_id`
- 标题
- 发布方
- 日期
- URL
- 置信度
- 关键事实
- 是否抓取全文

### 进入下一步条件

每个检索任务至少要返回可解释状态：

- 找到了来源。
- 没找到，但说明原因。
- 因缺少工具配置而跳过。
- 发现冲突，需要后续 Source QA 处理。

## 5. T3 — Source Pack 整理 & 来源质检

你给的写作流程里有 “Vault 检索 + Source Pack”。在 Search Agent 里，对应的是：

- 内部资料库 / 本地资料：如果有，会优先读。
- 原始联网结果：不能直接给 Writer 写报告。
- `CleanSourceList`：经过合并、去重、降噪、冲突检查后的 Source Pack。

### 你来做

一般不用操作。只有出现来源冲突时需要你选口径，例如：

- 一个来源说发布时间是 2026 年 5 月，另一个说 2026 年 6 月。
- 官方和媒体口径不一致。
- UGC 讨论很热，但没有官方或媒体印证。
- 金融数字来自不同口径。

### Codex 来做

SourceList Merger 先把所有来源合并去重，不新增事实。

Source QA Agent 再检查：

- 来源是否重复。
- 是否过旧。
- 是否缺 URL。
- 是否只有摘要或付费墙。
- 关键数字是否冲突。
- UGC 是否被错误当成事实。
- 是否需要补搜。

如果发现缺口，Gap Filler 只补 Source QA 指定的缺口，不能擅自扩题。

### 产出 Artifact

- `RawSourceList`
- `MergerLog`
- `SourceQANotes`
- `ConflictRegister`
- `GapList`
- `CleanSourceList`

### 进入下一步条件

没有未解决的高影响来源冲突；或者你明确选择采用哪个口径。

**闸门：来源冲突没有解决，不能进入正式分析。**

---

## 6. T4 — 结构化分析 & 专家判断

### 你来做

通常不用操作。你只需要等 Codex 产出分析结果。

### Codex 来做

Framework Analyst Agent 会把 `CleanSourceList` 转成 `ClaimGraph`。

简单说，`ClaimGraph` 就是“结论网络”：每条结论都要知道它来自哪个来源、属于事实/计算/假设/判断哪一类、置信度是什么。

如果题目包含营销、增长、用户、品牌、竞品：

- Marketing Specialist 会介入。
- 用 STP、4P、AARRR、JTBD、CBBE、Customer Journey 等框架分析。
- 营销 skill 用来增强分析方法，但不能凭空造事实。

如果题目包含财报、估值、市值、股价、收入：

- Finance Specialist 会介入。
- 用 KPI 树、杜邦、DCF、相对估值、现金流质量、风险专项等框架。
- 所有数字必须带期间、币种和口径。

### 会调用哪些 Skill / 工具

| 任务 | 可能调用 |
|---|---|
| 用户痛点 / JTBD | `customer-research` |
| 竞品档案 | `competitor-profiling` |
| 产品定位 | `product-marketing` |
| 定价 / 商业化 | `pricing` |
| 增长漏斗 | `analytics` / `ab-testing` |
| 品牌与传播 | `public-relations` / `social` |
| 财报解读 | `earnings-recap` / `estimate-analysis` |
| 估值 | `company-valuation` |
| 市场数据 | `yfinance-data` / `funda-data` |

### 产出 Artifact

- `ClaimGraph`
- `SpecialistNotes`
- `FinanceClaims`
- `MarketingClaims`

### 进入下一步条件

每条关键结论必须：

- 有 `source_id`。
- 标注事实/计算/假设/判断。
- 有置信度。
- 数字有口径。
- 不确定处不能写成确定结论。

---

## 7. T5 — 报告大纲 / 报告形态确认

你给的写作流程里有 “Writer 写大纲 + PM 审核大纲”。在 Search Agent 里，这一步不一定每次都单独停下来，但复杂报告建议停一下。

### 什么时候建议你确认大纲

- 要给领导或跨团队看。
- 要作为市场动作依据。
- 题目很大，比如“行业趋势 + 竞品格局 + 市场建议”。
- 你对报告形态有偏好。

### Codex 来做

Report Writer Agent 会根据读者和任务选择报告形态，而不是硬套一个模板。

常见形态：

| 报告形态 | 适用场景 |
|---|---|
| 竞品动态简报 | 竞品上新、功能变化、近期动作 |
| 市场响应建议 | 你想知道百度地图市场组怎么跟进 |
| 行业/政策影响简报 | 政策、宏观、行业变化 |
| Competitive Battlecard | 多竞品对比、销售/市场应对话术 |
| Finance / Investment Note | 财报、估值、投资判断 |
| Evidence Brief | 只查一个事实或数字 |

### 产出 Artifact

- `ReportPlan` 或报告大纲。
- 选定的报告形态。
- 核心判断草案。
- 章节安排。

### 进入下一步条件

简单报告可自动进入成稿；复杂报告建议你回复：

```text
大纲确认，继续写。
```

或：

```text
把报告形态改成市场响应建议，减少行业背景，多写我们能做什么。
```

---

## 8. T6 — 报告成稿

### 你来做

等待初稿。

### Codex 来做

Report Writer Agent 只能基于 `ApprovedClaimGraph` 写作，不能直接从原始网页里捞新事实，也不能把没有经过引用审计的内容写进正文。

报告必须做到：

- 先回答业务问题。
- 每个关键事实保留 `source_id`。
- 风险和不确定性显性写出。
- 不把资料堆成流水账。
- 不强行三段式或固定金字塔模板。
- 行动建议要有对象、场景、渠道、指标和验证方式。

### 产出 Artifact

- `ReportDraft`

---

## 9. T7 — AI 审核 & 引用审计

### 你来做

这一步通常先不用操作，等 Codex 展示审核摘要。

### Codex 来做

Citation Auditor Agent 会逐条检查：

- 报告里的事实有没有来源。
- 引用的来源是否真的支撑这句话。
- 有没有 Source Pack 之外的新事实。
- 付费墙/摘要来源有没有标注。
- UGC 有没有被误写成高置信度事实。

Humanizer Editor 会检查表达：

- 去掉模板腔。
- 去掉空泛转折。
- 去掉“显著、明显、用户普遍认为”等无来源泛化词。
- 保留数字、日期、引用、风险边界。

### 问题分级

| 等级 | 含义 |
|---|---|
| `must_fix` | 必须改，否则不能交付 |
| `should_fix` | 建议改，会影响质量 |
| `minor` | 可选润色 |

### 产出 Artifact

- `CitationAudit`
- `ApprovedClaimGraph`
- `HumanizerChangeLog`
- `AIReviewSummary`

### 进入下一步条件

Codex 展示稿件和 AI 审核摘要，等待你终审。

---

## 10. T8 — 你终审 & 修订

### 你来做

看报告和审核摘要，提出修改意见。

好的修改意见要具体：

```text
第 2 节对 B站反馈写得太确定。请改成低置信度观察，并说明样本局限。
```

```text
市场建议部分太泛，请按“目标人群、渠道、动作、指标”重写。
```

```text
把腾讯地图也加入竞品对比，但只补官方功能和媒体报道，不看 UGC。
```

不建议只说：

```text
再高级一点。
```

```text
写得更像人。
```

### Codex 来做

Codex 会把你的意见结构化成修订计划：

- 改哪里。
- 为什么改。
- 需要哪些来源支撑。
- 是否会引入新事实。
- 是否和 AI 审核意见冲突。

Rewriter 只能按修订计划改，不能擅自重写其他部分。

### 产出 Artifact

- `HumanReview`
- `RevisionPlan`
- `DraftV2`

### 进入下一步条件

修订计划没有未解决冲突；如果需要新增事实，Codex 会暂停请你确认是否补搜。

**闸门：修订计划涉及新事实或口径冲突时，你必须确认。**

---

## 11. T9 — 差异审核、终稿确认 & 存档

### 你来做

明确确认终稿。

```text
终稿确认，可以存档。
```

或者：

```text
终稿确认，不需要存档。
```

### Codex 来做

Integrity Diff Checker 会检查修订版有没有破坏证据链：

- 数字有没有变。
- 日期有没有变。
- `source_id` 有没有丢。
- 风险等级有没有被改高或改低。
- 低置信度内容有没有被写成确定事实。
- Key Message 是否仍然一致。

通过后，Codex 才能说终稿就绪。

如果你要求存档，Codex 生成独立存档文件，便于之后复盘。

### 产出 Artifact

- `IntegrityDiff`
- `FinalReport`
- `FinalApprovalSummary`
- `ArchiveSummary`（如你要求存档）

**闸门：AI 不能替你批准发布或归档。必须由你明确确认。**

## 12. 所有确认闸门汇总

| 闸门 | 触发条件 | 你需要做什么 |
|---|---|---|
| Brief / 审核卡片缺字段 | 主题、读者、输出类型、时间范围或调性不清 | 补充信息 |
| 审核卡片确认 | Step 0 输出框架、维度、关键词、来源范围后 | 回复确认或修改 |
| 来源冲突 | 官方、媒体、数据、UGC 口径冲突 | 选择采信口径或要求补搜 |
| 大纲/报告形态确认 | 复杂报告或你要求先看大纲 | 通过或提出修改 |
| AI 审核摘要 | 初稿和问题清单出来后 | 提出人工终审意见 |
| 修订计划冲突 | 修改需要新增事实或和旧口径冲突 | 确认是否补搜或采用新口径 |
| 终稿确认 | 存档或发布前 | 明确确认终稿 |

| Skill / 工具 | 中文说明 | 调用时机 |
|---|---|---|
| `marketing-ideas` | 营销方向发散 | T1 需求模糊、T4/T8 需要补市场动作 |

| `multi-search-engine` / realtime-search | 多搜索引擎 | T2 内部资料不足或需要联网搜索 |
| Firecrawl | 深度网页搜索和抓取 | T2 官方页、媒体页、JS 页面 |
| `finance-rss-reader` | 财经/产业 RSS | T2 近期新闻和动态扫描 |
| `agent-reach` / B站 CLI / opencli | 社媒和 UGC | T2 需要 B站、小红书、微博、知乎等公开讨论 |
| `competitor-profiling` | 竞品分析 | T4 竞品档案、功能对比 |
| `customer-research` | 用户研究/JTBD | T4 用户痛点、任务、场景 |
| `product-marketing` | 产品定位 | T4 STP/4P/定位分析 |
| `pricing` | 定价和商业化 | T4 商业化分析 |
| `analytics` / `ab-testing` | 指标和实验 | T4/T8 增长、漏斗、验证方案 |
| `copy-editing` | 编辑审查 | T7/T8 检查结构、事实、调性 |
| `humanizer` / `humanizer-zh` | 去 AI 味 | T7/T8 成稿和修订后 |
| `verification-before-completion` | 完成前核查 | T9 终稿就绪前 |

> 不是所有 skill 每次都会用。Codex 应该根据题目选择最小可用组合，不为了显得全面而堆工具。

## 14. B站和社媒怎么处理

B站、微博、小红书、知乎等社媒来源有价值，但不能当成硬事实。

它们适合回答：

- 用户有没有讨论某个功能。
- 讨论集中在哪些痛点。
- 竞品传播有没有形成声量。
- 市场组可以从哪些表达和内容角度切入。

它们不适合单独证明：

- 官方是否真的发布了某功能。
- 市占率、DAU、收入、留存等硬指标。
- 某个观点是否代表整体用户。

默认规则：

- UGC/Social Hunter 输出 `UGC###`。
- 默认 `confidence=low`。
- 只有被官方、媒体、应用商店或多平台重复信号印证后，才可以提高到中置信度。
- 不抓取隐私，不输出个人敏感信息。
- 不把单条评论写成“用户普遍认为”。

详细规则见 `references/social-ugc-policy.md`。

## 15. 报告质量标准

一份合格报告至少满足：

- 开头先回答业务问题，而不是先铺背景。
- 每个关键事实都有 `source_id`。
- 重要数字有时间、口径、来源。
- 风险和不确定性写出来。
- UGC 和媒体观点不冒充官方事实。
- 市场建议有对象、场景、渠道、指标、验证方式。
- 没有明显 AI 模板腔。

不合格表现：

- 只有资料汇总，没有判断。
- 只有判断，没有来源。
- 结论写得很满，但证据很弱。
- “加强宣传”“提升体验”这类空话太多。
- 所有报告都长得一样。

报告模板和质量线见 `references/report-templates.md`。

## 17. 常见问题

### 我需要在提示词里说“请使用 sub agent”吗？

不需要。你只要明确说“用 search-agent”，Codex 应该按项目里的多 Agent workflow 自动执行。

如果你担心它偷懒，可以补一句：

```text
严格按 search-agent 多 Agent workflow 执行，先审核卡片，确认后再进入检索。
```

### 如果我没有资料库怎么办？

可以直接说：

```text
我没有内部资料，请跳过 Vault，直接走公开搜索，但所有来源必须列 source_id。
```

Codex 会跳过内部资料阶段，进入联网检索。

### 如果 Firecrawl 或 B站工具不可用怎么办？

Codex 不能伪造结果。它应该明确标记：

- 哪个工具缺配置。
- 哪个节点被跳过。
- 用了什么兜底来源。
- 对报告置信度有什么影响。

### 为什么一定要先确认审核卡片？

因为调研最容易错的不是搜索，而是方向。  
如果一开始把“竞品动态”理解成“营销方案”，或者把“市场组建议”写成“中立行业报告”，后面搜得越多越浪费。

### 为什么 UGC 默认低置信度？

因为社媒讨论能说明“有人这样说”，但不能直接说明“市场就是这样”。  
它适合发现情绪、痛点、传播素材，不适合单独证明业务事实。

## 18. 给培训讲解时的一句话版本

Search Agent 的逻辑是：

> 先让人确认问题，再让 AI 分工找证据；先把来源洗干净，再让 AI 做分析；先审引用，再写报告；最后去 AI 味和做差异审核。人负责方向和批准，AI 负责检索、结构化、检查和成稿。
